* Overview

Phase 1 secured the registry door — only agents with trusted certificates can
register. But agent-to-agent calls are still plain HTTP. Any process on the
network that knows an agent's endpoint can call its MCP tools directly.

Phase 2 adds *mutual TLS to every agent-to-agent call*. Agents serve HTTPS
and require client certificates. The proxy layer presents certs when calling
other agents. Same root CA from Phase 1 — if your cert chains to a trusted
CA, you can talk to any agent in the mesh.

** Threat Model
- Rogue process on the network calls a legitimate agent's =/mcp= endpoint
- Man-in-the-middle intercepts agent-to-agent traffic
- Eavesdropping on tool call payloads (sensitive data in arguments/results)

** What Phase 2 Solves
- Agents reject MCP calls from untrusted clients (no valid cert = TLS handshake fails)
- All agent-to-agent traffic is encrypted (TLS)
- Mutual authentication — both caller and callee verify each other's identity

** What Phase 2 Does NOT Solve (Phase 3)
- Capability-level authorization (which agent can call which tools)
- Per-identity access policies
- Audit logging of inter-agent calls

** Key Design Decisions
- *Same certs as Phase 1* — no separate certificates for a2a communication
- *Same root CA = mesh-wide trust* — all agents with trusted certs can call all agents
- *No authorization rules* — Phase 2 is authentication only (are you in the mesh?)
- *SDK-native TLS* — each runtime uses its platform's native TLS mechanisms
- *Self-dependency skips TLS* — same-process calls use loopback, no network
- *Scheme in topology response* — registry returns =https://= endpoints when TLS is active
- *Backward compatible* — =MCP_MESH_TLS_MODE=off= (default) preserves plain HTTP behavior

* Implementation Notes

** Cert Reuse
Agents already have certificates from Phase 1:
- =MCP_MESH_TLS_CERT= — agent's certificate
- =MCP_MESH_TLS_KEY= — agent's private key
- =MCP_MESH_TLS_CA= — CA certificate (trust anchor)

These same files are used for:
1. Registration with registry (Phase 1 — already working)
2. Serving HTTPS on agent's HTTP port (Phase 2 — new)
3. Presenting client cert when calling other agents (Phase 2 — new)

** SDK HTTP Clients and Servers

| SDK        | HTTP Server      | HTTP Client | TLS Server Config            | TLS Client Config              |
|------------+------------------+-------------+------------------------------+--------------------------------|
| Python     | uvicorn/FastMCP  | httpx       | ssl_certfile, ssl_keyfile    | cert=(cert, key), verify=ca    |
| TypeScript | Node http/FastMCP| fetch       | https.createServer({cert..}) | https.Agent({cert, key, ca})   |
| Java       | Spring Boot/Tomcat| OkHttp3    | server.ssl.* properties      | SSLContext + SSLSocketFactory   |

** Endpoint URL Flow
#+begin_example
  1. Agent starts HTTPS server (when TLS mode != off)
  2. Agent registers with registry: endpoint = "https://hostname:port"
  3. Registry stores endpoint with https:// scheme
  4. Consumer agent receives dependency_available event with https:// endpoint
  5. Consumer proxy detects https:// → configures TLS on outgoing call
  6. mTLS handshake: both sides verify certs
  7. JSON-RPC POST /mcp proceeds over encrypted channel
#+end_example

** Self-Dependency Handling
#+begin_example
  Agent calls its own tool (self-dependency):
  - Endpoint resolves to localhost/127.0.0.1
  - Proxy detects loopback → uses plain HTTP
  - No TLS overhead for same-process calls
  - This is safe: the call never leaves the machine
#+end_example

** Java PEM-to-Keystore Challenge
Spring Boot requires PKCS12/JKS keystores, not raw PEM files.
Options:
1. Convert PEM to PKCS12 at startup using =java.security.KeyStore= API
2. Use Bouncy Castle for PEM parsing
3. Use Spring Boot 3.1+ =PemSslStoreDetails= (preferred if available)
