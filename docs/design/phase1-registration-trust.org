* Overview

MCP Mesh currently operates with open collaboration — any agent that knows the
registry URL can register and participate. Phase 1 introduces *entity-level
registration trust* so only agents belonging to operator-approved entities can
join the mesh.

** Threat Model
- Rogue process registers as an agent and gets discovered by legitimate agents
- Registry resolves rogue agent as a dependency → injected into legitimate workflows

** What Phase 1 Solves
- Registry validates agent identity at registration time
- Untrusted agents are rejected → never appear in dependency resolution
- Legitimate agents never discover or get injected with rogue agents

** What Phase 1 Does NOT Solve (Phase 2)
- Direct agent-to-agent HTTP calls (no mTLS yet)
- Rogue process calling a legitimate agent directly if it knows the address
- Man-in-the-middle on agent-to-agent communication

** Key Design Decisions
- *Entity-level trust*, not per-agent — operator registers entity CAs, entity manages its own agents
- *X.509 PKI* with entity-level intermediate CAs (RFC 5280 + RFC 8446)
- *Decorator stays clean* — no TLS config in mesh.agent/@MeshAgent, TLS is deployment config
- *No DB storage for CA certs* — trust anchors are infrastructure config (files, ConfigMap), not application data
- *No caching needed* — registry not in call path, validation only at registration
- *Cert rotation via existing heartbeat protocol* — 203 re-register signal, no new endpoints

** Architecture

*** Trust Flow
#+begin_example
  Operator registers Entity CA
          |
          v
  ┌──────────────────┐     ┌──────────────────────────┐
  │  Trust Backend    │     │  Credential Provider     │
  │  (Registry-side)  │     │  (Agent-side, Rust core) │
  │                   │     │                          │
  │  Validates cert   │ <── │  Presents cert at TLS    │
  │  chain against    │     │  handshake during        │
  │  trusted entity   │     │  registration POST       │
  │  CAs              │     │                          │
  └──────────────────┘     └──────────────────────────┘
#+end_example

*** PKI Hierarchy
#+begin_example
  Mesh Root CA (or operator-managed CA)
  ├── Entity A Intermediate CA

* Work Streams

** WS1: Trust Backend Abstraction (Go Registry) [7/7]

The registry needs a pluggable backend to validate agent certificates.
Registry is backend-agnostic — calls =backend.Verify(certChain)= and gets
back =(entityID, trusted)=.

*** Interface

#+begin_src go
type TrustBackend interface {
    // Verify validates a certificate chain and returns the entity identity.
    // Returns entity ID and nil error if trusted, error if rejected.
    Verify(certChain []*x509.Certificate) (entityID string, err error)

    // ListTrustedEntities returns all currently trusted entity identifiers.
    ListTrustedEntities() ([]TrustedEntity, error)

    // Name returns the backend identifier (for logging/config).
    Name() string
}
#+end_src

- Annotations for metadata: =mcp-mesh.io/entity-name= (display name), =mcp-mesh.io/expiry-warn-days= (expiry warning threshold), =mcp-mesh.io/contact= (entity contact)
- Watches for Secret changes via K8s informer → hot reload (not polling)

*** DONE WS1.5 — Implement backend chaining
- File: =src/core/registry/trust/chain.go=
- =MCP_MESH_TRUST_BACKEND=spire,k8s-secrets= — comma-separated, try in order
- First backend that trusts the cert wins
- Single backend is a chain of one — no special case
- Log which backend validated (for audit/debugging)

*** DONE WS1.7 — Implement SPIRE trust backend
- File: =src/core/registry/trust/spire.go=
- Validates certs against SPIFFE trust bundles keyed by trust domain
- Extracts entity ID from SPIFFE URI SANs (=spiffe://trust-domain/path=)
- =NewSPIREFromBundles()= constructor for testing without SPIRE SDK
- Live SDK integration deferred to =spire_connect.go= with =//go:build spire= tag

*** DONE WS1.6 — Integrate TrustBackend into registry server
- File: =src/core/registry/server.go=
- Initialize TrustChain from =MCP_MESH_TRUST_BACKEND= env var
- TLS listener with =ClientAuth: VerifyClientCertIfGiven= (allow non-TLS in =off= mode)
- Extract client cert from TLS connection state during POST /heartbeat
- Call =chain.Verify(certs)= → accept or reject registration
- Add =entity_id= field to agent record (from Verify response)
- When =MCP_MESH_TLS_MODE=off=, skip validation (backward compatible)
- *Dual-port listener*: Agent API on primary port (8000), admin API on separate port
  - Agent port: /heartbeat, /agents, /health (agent-facing)
  - Admin port: /admin/rotate, /admin/entities (operator-facing)
  - In K8s, NetworkPolicy restricts admin port to operator namespace only
  - Configured via =MCP_MESH_ADMIN_PORT= env var (default: disabled in dev, enabled in K8s)
- *Note*: Admin endpoint handlers (=/admin/entities=, =/admin/rotate=) are stubs — full implementation deferred to WS4

* Implementation Order

Priority order based on dependencies and value delivery:

1. *WS1.1 + WS1.3* — TrustBackend interface + FileStore (simplest production backend)
2. *WS2.1 + WS2.2 + WS2.4* — CredentialProvider + FileProvider + reqwest TLS config
3. *WS1.6* — Integrate into registry server (end-to-end registration with TLS)
4. *WS6.1 + WS6.2 + WS6.3* — Tests (validate the core flow works)
5. *WS1.2 + WS2.3 + WS3.4* — LocalCA + AutoProvider + =--tls-auto= (dev experience)
6. *WS3.1 + WS3.2 + WS3.3* — meshctl entity commands
7. *WS4.1 + WS4.2 + WS4.3* — Cert rotation via heartbeat
8. *WS6.4* — Rotation/eviction integration tests
9. *WS1.4 + WS1.5* — K8sSecrets backend + backend chaining
10. *WS5.1 + WS5.2 + WS5.3* — Helm chart integration
11. *WS7.1 + WS7.2* — Documentation

Steps 1-4 give a *working end-to-end TLS registration flow*.
Steps 5-6 give the *dev experience*.
Steps 7-8 give *operational tooling*.
Steps 9-11 are *production hardening*.

* Deferred (Explicitly Out of Phase 1)

These are referenced in env vars and design but deliberately not implemented in Phase 1:

- *SPIRE Provider* (=MCP_MESH_TLS_PROVIDER=spire=): Requires SPIRE infrastructure. Phase 1 gets FileStore + LocalCA working. SPIRE comes when K8s production deployments need it.
- *Vault Provider* (=MCP_MESH_TLS_PROVIDER=vault=): Similar — requires Vault infrastructure. FileProvider covers the same use case (Vault agent can write certs to disk, FileProvider reads them).
- *OPA Integration*: Complex policy engine for Phase 3 (authorization), not Phase 1 (registration trust).
- *Multi-cluster federation trust*: Future — requires cross-cluster CA trust distribution.
